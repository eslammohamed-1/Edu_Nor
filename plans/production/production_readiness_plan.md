# EduNor — خطة الجاهزية للإنتاج | Production Readiness Plan

## الهدف

تحويل المنصة من **واجهة + Mock/localStorage** إلى منتج **قابل للتشغيل في الإنتاج**: باكند حقيقي، مصادقة وصلاحيات من السيرفر، استضافة آمنة، ومسار واضح لكل مرحلة.

---

## الوضع الحالي (Baseline)

| المحور | الحالة |
|--------|--------|
| الواجهة (Vue 3 + Vite) | جاهزة نسبياً للطلاب والأدمن (مسارات `/admin/*`) |
| البيانات | Pinia + `localStorage`؛ لا مصدر حقيقي للحقيقة (Source of Truth) |
| المصادقة | Mock؛ توكن وهمي؛ سوبر أدمن من إعدادات `.env` |
| لوحة الإدارة | تعمل محلياً؛ الحماية على الواجهة فقط (`requiresSuperAdmin`) |
| التدقيق / النسخ الاحتياطي | محلي؛ لا استدامة مركزية |

---

## مبادئ تنفيذ

1. **السيرفر مصدر الحقيقة** لكل ما يخص المستخدمين، المحتوى، الاختبارات، الإعدادات، والتدقيق.
2. **عدم الثقة بالواجهة**: كل عمل حسّاس يُتحقق من الصلاحية على الـ API.
3. **HTTPS فقط** في الإنتاج؛ أسرار على السيرفر أو مدير أسرار (Secrets manager)، وليس في الريبو.
4. **نسخ احتياطي من قاعدة البيانات** وليس من `localStorage` فقط.

---

## المراحل المقترحة

### المرحلة P0 — قرارات وتجهيز (1–3 أيام)

**المخرجات**

- اختيار **Runtime باكند** (مثلاً: Node Nest/Fastify، Laravel، Django، أو BaaS مثل Supabase/Firebase — مع تقييم التكلفة والقيود).
- اختيار **قاعدة بيانات** (PostgreSQL موصى به للإنتاج).
- اختيار **استضافة الواجهة** (Cloudflare Pages، Netlify، VPS + Nginx، إلخ) و**استضافة API**.
- مستودع أو مشروع باكند (مجلد `server/` أو repo منفصل) + سياسة فروع ومراجعات.

**تعريف Done**

- وثيقة قرار (ADR مختصر) + مخطط نشر أولي (رسم تخطيطي نصي كافٍ).

---

### المرحلة P1 — أساس الباكند والمصادقة (1–3 أسابيع حسب الفريق)

**المهام**

- مشروع API مع routing صحي، طبقة **خدمات**، وطبقة **وصول للبيانات** (Repository أو ORM).
- جداول/كيانات أساسية: `users`, `roles`, `sessions` أو refresh tokens، وفصل **`super_admin`** عن الطالب والمدير.
- **تسجيل دخول**: bcrypt/argon2 لتخزين كلمات المرور؛ JWT قصير العمر + Refresh؛ أو جلسات HttpOnly حسب القرار.
- **Rate limiting** على `/auth/login`؛ تسجيل محاولات فاشلة (جدول أو SIEM لاحقاً).
- Endpoints صحة عامة: `GET /health`, إصدارات API تحت `/api/v1`.

**تعريف Done**

- Postman/OpenAPI يوثّق مسارات المصادقة الأساسية.
- اختبارات آلية للمسارات الحرجة (حتى لو عددها محدود في البداية).

---

### المرحلة P2 — مواءمة لوحة الإدارة مع API (2–4 أسابيع)

**استبدال تدريجي للـ stores المحلية**

| الواجهة الحالية | اتجاه الإنتاج |
|-----------------|---------------|
| `adminUsers` | CRUD مستخدمين + أدوار + حظر من API |
| `adminContent` | مواد، كورسات، فصول، دروس من API |
| `adminQuizzes` | اختبارات وأسئلة من API (+ استيراد JSON لاحقاً كمهمة فرعية) |
| `adminSettings` | إعدادات منصة مع صلاحية `settings:write` |
| `adminAudit` | قراءة سجل من API؛ الكتابة من السيرفر عند كل إجراء حسّاس |
| `adminSessions` | جلسات حقيقية من السيرفر |

**المهام**

- طبقة **`api client`** في الواجهة (fetch/axios) مع **اعتراض للتوكن** ومعالجة `401` → تحديث Refresh أو إعادة توجيه للّوجين.
- متغير بيئة **`VITE_API_BASE_URL`** للإنتاج والتطوير.
- الحفاظ على تجربة RTL والمسارات الحالية؛ تغيير **مصادر البيانات** فقط حيث يلزم.

**تعريف Done**

- السوبر أدمن يكمّل دورة كاملة: دخول → تعديل مستخدم → يظهر في DB → إعادة تحميل الصفحة يقرأ من السيرفر.
- لا اعتماد على `localStorage` لبيانات إدارية حرجة (يمكن الإبقاء على كاش اختياري لاحقاً).

---

### المرحلة P3 — مسار الطالب والمحتوى العام (بالتوازي أو بعد P2 حسب الأولوية)

- كورسات ودروس واختبارات للمستخدم المسجّل من API.
- احترام **سياسة المنصة** (صيانة، تسجيل مفتوح) من إعدادات السيرفر أو كاش قصير الأمد.

**تعريف Done**

- مسار سعيد كامل: تسجيل → تصفح محتوى مخوّل → درس → اختبار → حفظ محاولة على السيرفر.

---

### المرحلة P4 — النشر والبنية (أسبوع تقريباً بعد استقرار API)

- بناء الواجهة: `npm run build`؛ ضبط **`base`** في Vite حسب مسار النشر.
- خادم ويب أو CDN: **SPA fallback** إلى `index.html` لجميع المسارات بما فيها `/admin`.
- شهادة TLS، نطاق، وتوجيه `www`/`apex` إن لزم.
- متغيرات بيئة الإنتاج على منصة الاستضافة.

**تعريف Done**

- موقع حي على HTTPS؛ تحديث الصفحة على `/admin/users` لا يعطي 404.

---

### المرحلة P5 — تصلّب وتشغيل (Continuous)

- مراقبة أخطاء الواجهة (اختياري: Sentry).
- لوجات سيرفر وأتمتة نسخ احتياطي DB.
- سياسة تحديثات أمنية للتبعيات (`npm audit`، تحديث الإطار).

---

## ترتيب عمل مقترح للفريق الصغير

1. إنهاء **P0** في يوم واحد اجتماع + قرار مكتوب.
2. **P1** حتى يعمل Login حقيقي + مستخدم سوبر أدمن من seed على السيرفر.
3. **P2** صفحة صفحة: Users أولاً، ثم Content، ثم Quizzes، ثم Settings/Audit.
4. **P4** عند أول نسخة «Alpha» للاختبار الداخلي على سيرفر حقيقي.

---

## مخاطر يجب تجنبها

- نشر الواجهة الحالية كما هي مع الاعتماد على Mock وتسميتها «إنتاج».
- وضع أسرار السوبر أدمن أو مفاتيح JWT في الواجهة أو في Git.
- السماح لمسارات إدارية على API بدون تحقق دور `super_admin` على كل طلب.

| البند | الحالة |
|--------|--------|
| قرار ستاك مبدئي | مجلد `server/`: **Node + Fastify + Prisma + SQLite** (جاهز لاستبدال PostgreSQL لاحقاً) |
